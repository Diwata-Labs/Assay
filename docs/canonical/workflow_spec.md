# Workflow Specification


---

## 1. Test Run Workflow

### On-demand run (`assay run`)

1. Operator invokes `assay run [flags]`
2. CLI reads and validates `assay.toml` configuration
3. CLI pulls or verifies Playwright runner Docker image
4. CLI starts Docker container with target URL and suite parameters
5. Playwright executes configured test suite inside container
6. Container captures artifacts: screenshots, test logs, pass/fail results
7. Container exits; CLI collects artifacts from container output
8. Packet formatter converts artifacts into Grain-compatible task packets
9. Packets written as JSON files to output directory
10. CLI exits with code 0 (success) or 3 (test failures)

### Scheduled run

1. Scheduler fires based on registered cron expression
2. Same flow as on-demand run from step 2 onward
3. Schedule state updated: `last_run` timestamp and `last_result` recorded

---

## 2. SDK Ingest Workflow

1. Application embeds Assay browser SDK, initialized with API key + ingest endpoint URL
2. User or trigger invokes `assay.capture()` (or equivalent SDK method)
3. SDK captures:
   - Screenshot (browser canvas / DOM capture)
   - Page URL
   - Viewport dimensions
   - User agent string
   - ISO 8601 timestamp
   - Optional user-provided comment
4. SDK POSTs JSON payload to `/ingest` with `X-Assay-Key` header
5. FastAPI validates `X-Assay-Key` against local key store
6. **If invalid or missing:** HTTP 401 returned; payload rejected; no packet created
7. **If valid:** payload parsed and normalized; packet formatter invoked
8. Grain-compatible task packet written to output store
9. HTTP 200 returned to SDK caller

---

## 3. Task Packet Lifecycle

Packets generated by Assay are produced in `ready` status — immediately consumable by Grain or another downstream tool.

```
ready → (consumed by Grain or operator) → in_progress → review → done
```

Assay does not manage the packet lifecycle beyond initial creation. Downstream tools (Grain) own lifecycle transitions.

---

## 4. Output Format

- Format: JSON (Grain-compatible task packet schema)
- Location: configured output directory (`./assay-output` by default)
- Filename: `assay-<ISO-timestamp>-<uuid>.json`
- Schema: defined in `docs/canonical/data_contracts.md`
- One file per test run result or SDK capture event

---

## 5. Auth Enforcement

- All FastAPI endpoints under `/ingest` require a valid `X-Assay-Key` header
- Key validation occurs before any payload processing
- Invalid or missing key: HTTP 401; no partial processing
- Revoked keys: treated as invalid; HTTP 401

---

## 6. Docker Runner Contract

- Docker is required; there is no fallback to a host-native Playwright execution
- Each test run creates a fresh container — no container reuse between runs
- The runner Docker image must contain Playwright and all browser dependencies
- Containers are removed after artifact collection
- The runner Docker image tag is configured in `assay.toml` (`runner.docker_image`)

---

## 7. Scheduler Contract

- Scheduler state persists across CLI invocations (local filesystem)
- Scheduler does not require a persistent daemon process in v1
- Schedule execution is triggered by the CLI process when invoked, not a background service
- Missed runs (e.g., machine was off) are not automatically retried in v1

---

## 8. Configuration Precedence

In priority order:

1. CLI flags (highest)
2. Project-local `assay.toml`
3. Global `~/.assay/config.toml`
4. Built-in defaults (lowest)
