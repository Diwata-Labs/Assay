# Data Contracts

---

## 1. Assay Result Payload Schema

Assay produces this payload and writes it to the configured output directory after every run or capture. One file per run, named `assay-<ISO-timestamp>-<uuid>.json`.

Payloads are written as JSON files to the configured output directory. Filename: `assay-<ISO-timestamp>-<uuid>.json`.

### Required fields

```json
{
  "verification_id": "<string — stable ID assigned by Assay at submission time>",
  "task_id": "<string — the Grain task packet ID being verified, e.g. TASK-0070 — null for standalone runs>",
  "issue_type": "<enum: test_failure | bug_finding | screenshot_evidence | trace_capture | human_annotation>",
  "severity": "<enum: info | warning | error | critical>",
  "outcome": "<enum: pass | fail | inconclusive>",
  "summary": "<string — human-readable description of the finding>"
}
```

### Optional fields

```json
{
  "artifact_refs": ["<path or URI to captured artifact — screenshot, log, trace>"],
  "followup_candidates": [
    { "title": "<string>", "description": "<string>" }
  ],
  "verified_at": "<ISO 8601 datetime>"
}
```

### Field rules

- `verification_id`: UUID v4, generated by Assay at run/capture time; stable across retries
- `task_id`: the Grain `TASK-####` ID being verified; `null` when Assay runs in standalone mode (no Grain workflow)
- `issue_type`: maps to capture source — Playwright runner → `test_failure` or `bug_finding`; browser SDK → `screenshot_evidence` or `human_annotation`; trace capture → `trace_capture`
- `severity`: Playwright test failures default to `error`; browser SDK captures default to `info` unless user specifies otherwise
- `outcome`: `pass` = verification passed; `fail` = failure found; `inconclusive` = ambiguous result requiring human review
- `artifact_refs`: relative paths within the output directory or absolute URIs; may be empty
- `followup_candidates`: proposed follow-up work items surfaced to the Grain operator — never auto-created as packets
- `verified_at`: ISO 8601 timestamp of when Assay completed the verification run

### Delivery model

Assay writes the payload JSON to the configured output directory. The operator can inspect files directly or use `assay report` to view summaries. Files can be passed to any downstream tool that consumes the schema.

## 2. SDK Ingest Payload Schema

Payload POSTed by the TypeScript browser SDK to the `/ingest` endpoint.

```json
{
  "captured_at": "<ISO 8601 timestamp>",
  "url": "<full page URL>",
  "viewport": {
    "width": 1280,
    "height": 800
  },
  "user_agent": "<UA string>",
  "screenshot": "<base64-encoded PNG string>",
  "user_comment": "<optional string — omit or null if none>",
  "metadata": {}
}
```

**HTTP headers (required):**

| Header | Value |
|--------|-------|
| `X-Assay-Key` | Valid API key |
| `Content-Type` | `application/json` |

**Validation rules:**
- `captured_at`: required; must be valid ISO 8601
- `url`: required; non-empty string
- `viewport`: required; both `width` and `height` must be positive integers
- `user_agent`: required; non-empty string
- `screenshot`: required; valid base64-encoded PNG
- `user_comment`: optional; `null` or omitted is valid
- `metadata`: optional object; additional fields accepted but not required

## 3. API Key Store Schema

Local file: `~/.assay/keys.json`

```json
{
  "version": "1",
  "keys": [
    {
      "id": "<uuid-v4>",
      "label": "<human-readable label>",
      "hash": "<bcrypt hash of raw key>",
      "created_at": "<ISO 8601 timestamp>",
      "revoked": false,
      "revoked_at": null
    }
  ]
}
```

**Security rules:**
- Raw key values are never stored — only bcrypt hashes
- Raw key is printed exactly once at `assay key create` time
- `revoked: true` entries must be preserved (not deleted) for audit trail
- `revoked_at`: ISO 8601 timestamp set when key is revoked; `null` if active

## 4. Schedule State Schema

Local file: `~/.assay/schedules.json`

```json
{
  "version": "1",
  "schedules": [
    {
      "id": "<uuid-v4>",
      "cron": "<5-field cron expression>",
      "suite": "<suite name>",
      "target": "<url or null to use config default>",
      "created_at": "<ISO 8601 timestamp>",
      "last_run": "<ISO 8601 timestamp or null>",
      "last_result": "success | failure | null"
    }
  ]
}
```

## 5. Configuration File Schema (`assay.toml`)

```toml
[project]
name = "<string>"           # required

[runner]
docker_image = "<string>"   # required; e.g. "assay-playwright:latest"
timeout_seconds = 300       # optional; default 300

[output]
directory = "<path>"        # optional; default "./assay-output"

[serve]
host = "<string>"           # optional; default "127.0.0.1"
port = 8000                 # optional; default 8000

[keys]
store = "<path>"            # optional; default "~/.assay/keys.json"

[schedule]
store = "<path>"            # optional; default "~/.assay/schedules.json"
```

### CLI commands

| Command | Description |
|---------|-------------|
| `assay run` | Execute a test run using the Playwright + Docker runner |
| `assay serve` | Start the FastAPI ingest server |
| `assay report` | Display or export task packet summaries |
| `assay key create` | Generate a new API key |
| `assay key list` | List all API keys (IDs and labels only) |
| `assay key revoke` | Revoke an API key by ID |
| `assay schedule add` | Register a new scheduled run |
| `assay schedule list` | List all active schedules |
| `assay schedule remove` | Remove a schedule by ID |
| `assay schedule run` | Start the foreground scheduler loop (runs until Ctrl+C) |

---

## 6. Contract Versioning

- `schema_version` in task packets tracks the output schema
- Breaking schema changes (removed fields, changed types) require a major version bump
- Additive changes (new optional fields) are non-breaking and do not require a bump
- The key store and schedule state schemas track `version` independently
- All schema versions follow semver
